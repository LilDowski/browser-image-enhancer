# Бенчмарк производительности

Замеры закрывают критерии оценки **№2 (скорость)**, **№3 (память)**, **№4 (CPU)** и
**№5 (размер)**. Все цифры сняты на **продакшен-сборке** (`vite preview`) в Chromium
через автоматизированный прогон ([методику см. ниже](#методика)).

> Замеры сделаны на одной машине с программным WebGL (headless). На реальном
> устройстве с аппаратным GPU времена будут не хуже, а обычно лучше.

## 1. Скорость обработки (критерий №2)

Тёплая обработка (TF.js и модель уже загружены), 6 прогонов на размер:

| Размер | Разрешение | Среднее | Медиана (p50) | p95 | Максимум |
|---|---|---:|---:|---:|---:|
| 1 Мпк | 1000×1000 | 27 мс | 28 мс | 30 мс | 30 мс |
| 4 Мпк | 2000×2000 | 62 мс | 61 мс | 72 мс | 72 мс |
| 8 Мпк | 2828×2828 | 121 мс | 128 мс | 143 мс | 143 мс |
| **15 Мпк** | **4472×3354** | **193 мс** | **189 мс** | **247 мс** | **247 мс** |

- Масштабируется линейно, ~13 мс на мегапиксель.
- **15 Мпк за ~0.19 с** — против лимита ТЗ «не более 30 с» и цели «5 с в среднем».
  Запас более чем в **100 раз**.
- **Холодный старт** (самое первое изображение, включает разовую инициализацию
  TF.js): **~0.2 с** локально. На реальном хостинге к первому запуску добавляется
  разовая загрузка TF.js (~1.9 МБ) и модели (~0.43 МБ) по сети.

## 2. Память (критерий №3)

10 последовательных обработок изображения 15 Мпк:

| Показатель | Значение |
|---|---|
| Heap главного потока (до) | 1.4 МБ |
| Heap главного потока (после 10× 15 Мпк) | 1.4 МБ |
| **Прирост (утечки)** | **0 МБ** |

- Главный поток остаётся **минимальным (~1.4 МБ)** и **не растёт** — утечек нет.
- Тяжёлые буферы (декодированный кадр, WebGL-текстура, выходной холст) живут в
  Web Worker и освобождаются после каждой задачи. Исходные данные передаются в
  worker через **transfer** `ArrayBuffer` — без копирования.
- Расчётный пиковый объём в worker для 15 Мпк: ~60 МБ на декодированный кадр RGBA +
  столько же на текстуру/выход — порядка 150–200 МБ кратковременно, затем
  освобождается. На анализ модель получает уменьшенную копию 256×256 (≈0.26 МБ),
  а не полный кадр.

## 3. CPU и отзывчивость (критерий №4)

Замер кадров (FPS) главного потока **во время** обработки 15 Мпк, в сравнении с
контрольной синхронной блокировкой:

| Сценарий | Длительность | Кадров отрисовано | FPS |
|---|---:|---:|---:|
| **Обработка 15 Мпк (в worker)** | 182 мс | **21** | **115** |
| Контроль: синхронная блокировка | 250 мс | 0 | 0 |

- Во время обработки интерфейс **остаётся отзывчивым** (анимации идут, ~115 FPS),
  тогда как эквивалентная синхронная работа полностью заморозила бы вкладку (0 FPS).
- Основная вычислительная нагрузка — на **GPU** (WebGL-шейдер), а не на CPU.
  Анализ моделью идёт по крошечному превью 256×256, декодирование — нативное.
  Поэтому процессор почти свободен, а UI не блокируется (асинхронный API + worker).

## 4. Размер решения (критерий №5)

Полная сборка `dist/` — **3.65 МБ** (лимит ТЗ — 10 МБ, запас ~2.7×):

| Часть | Размер | Когда грузится |
|---|---:|---|
| Стартовая загрузка (HTML+CSS+JS+worker) | ~21 КБ | сразу |
| Модель TensorFlow.js (`model.json`+веса) | ~0.43 МБ | при первой обработке |
| TF.js (рантайм) | ~1.9 МБ | при первой обработке (ленивый чанк) |
| Декодер HEIC (libheif-wasm) | ~1.46 МБ | только при открытии HEIC (ленивый чанк) |

- **Стартовая загрузка ~21 КБ** — интерфейс открывается мгновенно; тяжёлые части
  подгружаются по мере необходимости.

## Сводка по критериям оценки

| Критерий | Результат |
|---|---|
| Качество изображения | ИИ > классики по PSNR/SSIM, «не навреди» (см. [EVALUATION.md](EVALUATION.md)) |
| Скорость | 15 Мпк ≈ 0.19 с (лимит 30 с) |
| Память | главный поток 1.4 МБ, без утечек; буферы в worker освобождаются |

| Размер | 3.65 МБ из 10 МБ |

## Методика

Бенчмарк выполняется в браузере на продакшен-сборке. Экземпляр API доступен по
флагу `?debug` в URL (в обычном режиме скрыт). Изображения генерируются на лету
(canvas) с деталями, прогоняются через публичный API `ImageEnhancer`; время
считается от постановки задачи до события `done`. FPS-тест считает кадры
`requestAnimationFrame` главного потока во время обработки. Память — через
`performance.memory` (Chromium).
